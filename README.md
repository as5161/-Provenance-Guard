# Provenance Guard

A backend system that creative platforms can integrate to assess whether
submitted text was written by a human or generated by AI. It analyzes content
with two independent detection signals, combines them into a single confidence
score, surfaces a plain-language transparency label, lets creators appeal
classifications, rate-limits abuse, and records every decision in a structured
audit log.

Because wrongly labeling a human's work as AI-generated is the more harmful error
on a writing platform, the system is deliberately designed to express uncertainty
honestly rather than force a confident verdict.

---

## Architecture Overview

A single submission travels through the system as follows:

1. A creator sends text and a creator ID to `POST /submit`.
2. The Flask app validates the input and generates a unique `content_id`.
3. The text is sent to **Signal 1 (Groq LLM)**, which judges it semantically, and
   to **Signal 2 (stylometric analyzer)**, which measures structural properties.
4. Both signal scores are passed to the **confidence-scoring function**, which
   combines them into a single score and assigns an attribution category.
5. The score selects one of three **transparency labels**.
6. The full result — both signal scores, combined confidence, attribution, label,
   timestamp, status — is written to the **SQLite audit log**.
7. The response (content_id, attribution, confidence, label) is returned.

If a creator contests a verdict, `POST /appeal` looks up the record by
`content_id`, flips its status to `under_review`, attaches the creator's
reasoning, and logs the appeal alongside the original decision.

```
POST /submit { text, creator_id }
        |
        v
   Flask app  --> generates content_id
        |
        +-----------------------+
        v                       v
  Signal 1: Groq          Signal 2: Stylometric
  (semantic)              (structural)
        |                       |
        +-----------+-----------+
                    v
          Confidence scoring (weighted average + disagreement guard)
                    v
            Label generator (3 variants)
                    |
            +-------+-------+
            v               v
        Audit log     Response (JSON)
```

### API Endpoints

| Method | Endpoint     | Purpose                                            |
|--------|--------------|----------------------------------------------------|
| POST   | `/submit`    | Analyze text; returns content_id, attribution, confidence, label |
| POST   | `/appeal`    | Contest a classification; flips status to under_review |
| GET    | `/log`       | Returns recent audit-log entries as JSON           |
| GET    | `/dashboard` | Visual analytics page (stretch feature)            |

## Detection Signals

The pipeline uses two genuinely independent signals — one semantic, one
structural — so that each covers the other's blind spots.

### Signal 1 — Groq LLM (semantic)

Sends the text to `llama-3.3-70b-versatile` with a prompt asking it to judge
whether the writing reads as human or AI-generated, returning a score from 0.0
(clearly human) to 1.0 (clearly AI). This captures meaning, tone, and stylistic
coherence holistically.

**What it misses:** It is an informed opinion, not a calibrated instrument. It can
be confidently wrong, is fooled by lightly-edited AI text, and may unfairly lean
toward "AI" for humans who write in a polished, formal style.

### Signal 2 — Stylometric heuristics (structural)

Pure-Python analysis with no external libraries. It computes two metrics:

- **Sentence-length variance** — how much sentence lengths vary. Human writing is
  bursty and irregular (high variance); AI tends toward uniform sentence lengths
  (low variance), so low variance leans AI.
- **Type-token ratio (vocabulary diversity)** — unique words divided by total
  words. Combined with variance, low diversity adds weight toward AI.

These are converted into a single 0.0–1.0 score (variance weighted 0.65, TTR 0.35).
A short-text guard returns 0.5 ("can't tell") when there isn't enough data to
measure meaningfully.

**What it misses:** It ignores meaning entirely. Deliberately uniform human writing
(legal briefs, academic abstracts) or a poem built on repeated structure can read
as "AI" to it, and lexically-rich AI text can read as "human."

**Why this pairing:** The two signals look at different properties, so their
*disagreement* is itself informative — it tells the system to be less confident,
which is exactly the behavior that protects human writers from false accusations.

## Confidence Scoring

The two signal scores are combined into a single confidence score (0.0–1.0):

1. **Weighted average**, LLM-heavy: `0.7 * llm_score + 0.3 * stylo_score`. Groq is
   the primary detector; stylometrics is a corroborating second opinion.
2. **Disagreement guard:** if the two signals differ by more than 0.4, the result
   is pulled halfway toward 0.5. Strong disagreement lowers confidence rather than
   splitting the difference into a confident verdict — this is the system's
   core protection against false positives.

The combined score maps to an attribution category:

| Score range | Attribution    |
|-------------|----------------|
| >= 0.65     | likely_ai      |
| 0.45 – 0.64 | uncertain      |
| < 0.45      | likely_human   |

### How it was validated

The scoring was tested against a panel of deliberately chosen inputs spanning the
range: clearly AI, clearly human, formal human (a known false-positive trap), and
lightly-edited AI. Scores varied meaningfully across them and — critically — no
human-written sample was ever pushed into the `likely_ai` category.

### Two example submissions (different confidence levels)

**Higher-confidence case (clearly human, casual review):**
```
text:       "ok so i finally tried that ramen place downtown and honestly?
             underwhelming. broth was fine but way too salty..."
llm_score:  0.2
stylo_score: 0.216
confidence: 0.205   ->  likely_human
```

**Lower-confidence case (formal human prose — the false-positive trap):**
```
text:       "The relationship between monetary policy and asset price inflation
             has been extensively studied..."
llm_score:  0.7
stylo_score: 0.416
confidence: 0.615   ->  uncertain
```

In the second case, Groq leaned toward AI (0.7) while the stylometric signal did
not. Their disagreement engaged the guard, pulling the score into "uncertain"
instead of falsely flagging a human writer as AI. This is the asymmetry the system
is built around: when in doubt, it declines to accuse.

## Transparency Label

The label shown to a reader changes based on the attribution. All three variants:

**High-confidence AI (`likely_ai`):**
> "This content shows strong signals of AI generation. Our system assessed it with
> high confidence, but detection is not perfect — you are welcome to appeal with
> factual evidence."

**High-confidence human (`likely_human`):**
> "This content reads as human-written. Our system found no strong signals of AI
> generation."

**Uncertain (`uncertain`):**
> "Our system could not determine with confidence whether this was written by a
> human or AI. Treat this result as inconclusive."

All three were verified reachable by submitting inputs that produce each
confidence level. The AI label deliberately includes an invitation to appeal,
reflecting the design priority that a false accusation should always have a
remedy.

## Appeals Workflow

Any creator can contest a classification via `POST /appeal` with a `content_id`
and `creator_reasoning`. The endpoint:

1. Validates that a submission with that `content_id` exists (returns 404 if not).
2. Updates the record's status to `under_review`.
3. Attaches the creator's reasoning to the record.
4. Logs the appeal alongside the original classification decision.
5. Returns a confirmation.

A human reviewer opening the record sees the original text, both signal scores, the
combined confidence, the verdict, and the creator's reasoning side by side.
Automated re-classification is out of scope by design — the appeal opens a review,
it does not auto-overturn the verdict.

## Rate Limiting

Implemented with Flask-Limiter on the `/submit` endpoint, tracked per IP address:

```
@limiter.limit("10 per minute;100 per day")
```

**Reasoning for these values:**

- A legitimate creator submits their own work. Even a prolific writer revising and
  resubmitting rarely exceeds a handful of submissions in a sitting, and
  essentially never 10+ in a single minute. **10/minute** comfortably covers real
  use without ever obstructing a genuine user.
- An adversary (a script flooding the system, or someone probing the detector with
  many variations) wants hundreds or thousands of fast requests. 10/minute throttles
  that hard, and the **100/day** ceiling stops a slow-drip flood that stays under the
  per-minute cap.
- Each submission triggers a real Groq API call, so rate limiting also protects the
  system from runaway cost under abuse.

**Verified behavior** (12 rapid requests to `/submit`):
```
Request 1  : 200
Request 2  : 200
...
Request 10 : 200
Request 11 : 429
Request 12 : 429
```
The first 10 succeed; the 11th onward are rejected with HTTP 429 (Too Many Requests).

captures: timestamp, content_id, creator_id, both individual signal scores
(`llm_score`, `stylo_score`), combined confidence, attribution, status, and

## Stretch Feature: Analytics Dashboard

`GET /dashboard` returns a visual HTML page summarizing detection patterns across
all submissions. It reads the SQLite database with `COUNT`, `GROUP BY`, and `AVG`
queries and displays:

- **Total submissions**
- **Attribution breakdown** (likely_ai / uncertain / likely_human) as colored bars
- **Appeal rate** (appeals filed ÷ total submissions)
- **Average confidence score** (the additional chosen metric)

This gives a platform operator a quick read on how the detector is behaving and how
often creators contest its verdicts.

## Known Limitations


The clearest limitation follows directly from the design of the two signals:

**Lightly-edited AI text is frequently missed.** When AI output is touched up by a
human, it loses the structural uniformity the stylometric signal looks for and
reads more naturally to Groq. In testing, a lightly-edited AI passage scored
`llm=0.4, stylo=0.0 -> confidence 0.28 (likely_human)` — a false negative. This is
a deliberate tradeoff: catching such text would require making the system more
aggressive, which would raise false positives against skilled human writers. On a
writing platform, a false positive (accusing a human) is the worse error, so the
system accepts more false negatives in exchange for protecting human writers.

A second limitation: **lexically-rich AI text** (formal, varied, sophisticated
vocabulary) does not trigger the stylometric signal at all, because that signal
only detects structural uniformity and repetition. The stylometric signal returned
0.0 across an entire test panel of varied texts, confirming it fires only on a
narrow band of structurally-uniform writing and otherwise defers to Groq.

---

## Spec Reflection


**One way the spec helped:** Writing the confidence-scoring and label thresholds in
`planning.md` *before* coding meant the implementation had concrete targets to hit.
Deciding what 0.5 should mean to a user, and where the category boundaries sat,
turned a vague "combine the signals somehow" into a specific, testable function.

**One way the implementation diverged:** The plan treated the stylometric signal as
a roughly co-equal detector. In practice, testing revealed it only fires on
structurally-uniform text and sits near 0.0 for most varied writing. Rather than
tune it more aggressive (which would have caused false positives), the design
shifted to treating stylometrics as a conservative human-protector that defers to
Groq — a divergence driven by what the test data actually showed.

## AI Usage


**Instance 1 — Stylometric signal generation.** I directed an AI tool to generate
the `stylometric_signal` function from my planning spec (sentence-length variance +
type-token ratio, output 0–1). It produced a working function with threshold
constants. On testing, it scored an entire panel of texts at 0.0, which prompted me
to investigate rather than accept it blindly; I traced the arithmetic and confirmed
the all-zeros was an honest property of the signal, not a bug, and documented it as
a known limitation instead of over-tuning to hide it.

**Instance 2 — Confidence-scoring logic.** I directed the AI to implement the
combining function matching my planning thresholds. I verified the generated code
against my specified ranges and confirmed the disagreement-guard behaved as
intended (pulling strongly-disagreeing signals toward 0.5). I kept the 0.7/0.3
weighting as a deliberate "Groq is primary" choice after reviewing how an even
split would have dragged scores down given the stylometric signal's conservative
behavior.

## Running the Project

```bash
python -m venv .venv
source .venv/Scripts/activate      # Windows Git Bash
# or: .venv\Scripts\Activate.ps1   # Windows PowerShell
pip install -r requirements.txt
```

Create a `.env` file with your Groq key (never commit this):
```
GROQ_API_KEY=your_key_here
```

Run the server:
```bash
python app.py
```

The API is served at `http://127.0.0.1:5000`.
